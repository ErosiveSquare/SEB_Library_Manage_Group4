# 1.1采访子系统DFD

这是**采访子系统（1.1）的DFD（数据流程图）**，以下是基于第二张图片内容的准确且全面的描述：

### 一、外部实体（共2个）

外部实体是该子系统的交互对象，包括：

1. **读者**：
   - 向「1.1.1 采访验收」提供“荐购信息”（即读者希望图书馆购买的书籍信息）。
2. **采编部**：
   - 向「1.1.2 订单管理」提供业务指令或审核信息；
   - 向「1.1.4 退货」提供“退货原因”及“经手人”信息。

### 二、处理过程（4个处理模块）

该子系统细分为4个具体的处理步骤：

1. **1.1.1 采访验收**：
   - **输入**：接收读者的“荐购信息”；
   - **交互**：与“采访清单”进行双向数据交互（可能涉及查重、写入新荐购记录）；
   - **流向**：将处理后的信息传递给「1.1.2 订单管理」。
2. **1.1.2 订单管理**：
   - **输入**：接收来自「1.1.1」的数据流，以及采编部的操作；
   - **交互**：
     - 读取“字典”信息（用于校验书商、出版社等基础信息）；
     - 与“订单信息表”进行双向数据交互（生成或更新订单）；
   - **流向**：将订单数据传递给「1.1.3 验收」。
3. **1.1.3 验收**：
   - **输入**：接收来自「1.1.2」的订单流；
   - **输出**：将合格的书籍信息写入“验收清单”；
   - **异常流向**：若书籍“不合格”，数据流向「1.1.4 退货」。
4. **1.1.4 退货**：
   - **输入**：
     - 接收来自「1.1.3」的“不合格”书籍信息；
     - 接收采编部提供的“退货原因、经手人”信息；
   - **输出**：将退货记录写入“退货清单”。

### 三、数据存储（5个）

该子系统涉及的数据存储表如下：

1. **采访清单**：与「1.1.1 采访验收」双向交互，存储读者荐购的原始记录。
2. **订单信息表**：与「1.1.2 订单管理」双向交互，存储正式生成的采购订单数据。
3. **字典**：单向输入给「1.1.2 订单管理」，提供基础参照数据（如出版社、书商信息）。
4. **验收清单**：接收「1.1.3 验收」的输出，存储验收合格入馆的图书记录。
5. **退货清单**：接收「1.1.4 退货」的输出，存储因质量或版本问题退回书商的记录。

### 四、核心逻辑

该DFD展示了图书从**“读者需求”**转化为**“馆藏实体”**的前半段全过程：

- **需求转化**：流程始于读者的荐购，经过“采访验收”转化为待处理清单。
- **订购流程**：通过“订单管理”结合基础字典生成正式订单。
- **质检分流**：书籍到货后的“验收”环节是关键分歧点——合格品进入“验收清单”（随后进入编目），不合格品进入“退货”流程并记录原因。

# 1.2编目子系统DFD描述

这是**编目子系统（1.2）的DFD（数据流程图）**，以下是基于第三张图片内容的准确且全面的描述：

### 一、外部实体（共1个）

该子系统主要由内部人员操作，仅涉及一个外部实体：

1. **采编部**：
   - 向「1.2.1 取采访」提供“图书信息”；
   - 向「1.2.7 编目查询」提供“编目信息”（查询条件）；
   - 向「1.2.6 注销报损」提供“图书信息、经手人、销毁原因”。

### 二、处理过程（7个处理模块）

该子系统逻辑较为复杂，包含从获取书籍到入库上架的全流程：

1. **1.2.1 取采访**：
   - **输入**：接收采编部的“图书信息”；
   - **交互**：读取“验收清单”中的数据（承接上一环节）；
   - **流向**：
     - 将“查重前编目信息”传递给「1.2.2 编目查重」；
     - 若信息有误，产生“无效信息”反馈。
2. **1.2.2 编目查重**：
   - **输入**：接收来自「1.2.1」的待查信息；
   - **交互**：读取“期刊库表”和“图书流通库表”，检查馆内是否已有相同书目；
   - **流向**：将“更新后编目信息”传递给「1.2.3 编目」。
3. **1.2.3 编目**：
   - **输入**：接收查重后的信息；
   - **输出**：生成标准的编目数据，写入“编目清单”。
4. **1.2.4 移送**：
   - **输入**：读取“编目清单”中的已编目数据；
   - **输出**：将正式数据分别写入/更新到“期刊库表”和“图书流通库表”，完成入库。
5. **1.2.5 新书通报**：
   - **输入**：读取“编目清单”数据；
   - **输出**：提取新书摘要写入“新书通报表”，供Web端展示。
6. **1.2.6 注销报损**：
   - **输入**：接收采编部的“图书信息、经手人、销毁原因”；
   - **交互**：在“期刊库表”或“图书流通库表”中标记或删除对应书籍；
   - **输出**：生成记录写入“报损记录”。
7. **1.2.7 编目查询**：
   - **输入**：接收采编部的查询请求；
   - **交互**：检索“编目清单”以获取当前编目状态。

### 三、数据存储（6个）

该子系统涉及多个核心数据库表：

1. **验收清单**：作为输入源，提供已验收合格的书籍数据。
2. **编目清单**：核心临时存储，连接编目、移送、通报和查询四个环节。
3. **期刊库表**：
   - 参与查重（读）；
   - 接收移送入库（写）；
   - 参与注销报损（读/写）。
4. **图书流通库表**：
   - 参与查重（读）；
   - 接收移送入库（写）；
   - 参与注销报损（读/写）。
5. **新书通报表**：接收「1.2.5」的输出，存储新书资讯。
6. **报损记录**：接收「1.2.6」的输出，存储图书销毁或报废的流水记录。

### 四、核心逻辑

该DFD展示了图书进入馆藏系统的核心**“身份确认”**过程：

- **查重机制**：在正式编目（1.2.3）前，强制进行查重（1.2.2），确保馆藏数据的唯一性和准确性，避免重复建档。
- **缓冲设计**：使用“编目清单”作为缓冲区，编目人员先在此完成工作，确认无误后通过“移送”（1.2.4）才正式进入流通库，保证了流通数据的纯净。
- **并行处理**：新书入库后，自动并行触发“入库移送”和“新书通报”，提高了上架效率。
- **逆向流程**：独立设计了“注销报损”流程，处理图书的全生命周期结束（剔除、丢失等）。



# 1.3流通子系统

这是一张描述 **1.3 流通子系统** 的数据流图（DFD）。为了帮助你全面理解该系统的逻辑，我将从**外部实体**、**数据存储**、以及具体的**处理过程（加工）**三个维度进行详细描述。

### 1. 系统概述

该图展示了图书馆管理系统中“流通子系统”的运作流程。主要的功能是由**流通部**发起的图书借阅、归还以及配置管理，同时系统后台还处理图书预约和到期检查等逻辑。

------

### 2. 主要组成元素

#### **A. 外部实体 (External Entity)**

- **流通部**：这是系统的主要交互方（数据源），负责发起借书、还书和修改配置的动作。

#### **B. 数据存储 (Data Stores)**

系统涉及以下数据表，用于存储和检索信息：

1. **预约信息表**：存储读者的预约请求状态。
2. **借阅记录**：存储图书被借出的历史和当前状态。
3. **读者信息表**：存储读者的基本信息及借阅权限。
4. **图书流通库表**：存储图书的库存及在馆/借出状态。
5. **默认处罚配置**：存储逾期罚款规则等配置参数。
6. **处罚记录**：存储因逾期或违规产生的罚款记录。

------

### 3. 详细处理流程描述

该DFD图包含5个核心处理过程（Process），具体的数据流向如下：

#### **1.3.1 借书 (Borrow Book)**

这是流通部的核心业务之一。

- **输入流**：
  - 来自**流通部**的“借书信息”（如读者ID、图书ID）。
- **数据交互（读/写）**：
  - **预约信息表**：系统检查该书是否被他人预约，或更新该读者的预约状态。
  - **借阅记录**：生成新的借书记录。
  - **读者信息表**：验证读者身份、借阅权限，并更新读者当前借阅数量。
  - **图书流通库表**：查询图书状态（是否在架），并将图书状态更新为“借出”。

#### **1.3.5 修改配置信息 (Modify Configuration)**

这是一个管理性质的功能。

- **输入流**：
  - 来自**流通部**的“配置信息”。
- **数据交互**：
  - **写入**到 **默认处罚配置**：更新罚款规则、借阅期限等系统参数。

#### **1.3.2 还书 (Return Book)**

处理图书归还及可能的违规处理。

- **输入流**：
  - 来自**流通部**的“还书信息”。
- **数据交互**：
  - **读取** **默认处罚配置**：获取判断是否逾期的标准。
  - **交互** **读者信息表**：更新读者状态（如恢复借阅名额）。
  - **交互** **借阅记录**：更新记录为“已还”，计算借阅时长。
  - **交互** **图书流通库表**：将图书状态更新为“在馆”。
  - **写入** **处罚记录**：如果发现逾期或图书损坏，生成相应的处罚记录。

#### **1.3.3 图书预约 (Book Reservation)** & **1.3.4 预约到期检查 (Reservation Expiry Check)**

这两个过程位于图的底部，且有交互关系，主要处理非现场的预约逻辑。

- **1.3.3 图书预约**：
  - 与 **预约信息表** 和 **图书流通库表** 进行双向交互（验证库存并锁定图书）。
  - **输出流**：“非法信息”（可能是指预约失败或不符合规则的反馈）。
  - 与 **1.3.4** 存在数据交互。
- **1.3.4 预约到期检查**：
  - 定期检查 **预约信息表** 和 **图书流通库表**。
  - 如果预约超期未取，可能会释放图书库存，并向 **处罚记录** 写入数据（记录爽约惩罚）。

------

### 4. 总结

这张DFD图清晰地表达了：

1. **流通部** 是操作的主体。
2. **借书** 和 **还书** 是涉及数据表最多的复杂业务，需要同时同步读者、库存和记录表。
3. **配置管理** 独立于业务流，直接影响系统的规则参数。
4. **预约管理** 包含自动化的检查机制（到期检查），并能自动触发处罚逻辑。

### **3. 数据处理过程与流向详解 (Process & Data Flows)**

#### **A. 读者信息管理模块 (左侧)**

该模块由“读者和管理员”实体发起，针对**读者信息表**进行操作。

- **1.4.2 录入读者信息**

  - **输入**：实体提供“读者编号”。
